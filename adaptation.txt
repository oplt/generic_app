# Repository Improvement Report

Generated from deep repository analysis (backend modules, frontend, AI/RAG/memory stack, caching, tests, ADR 0003). Priority items 1–5, 10, frontend UX (#4), performance, caching, ADR 0003 boundary ports, UI/UX pass, RAG algorithm fixes, and dead-code cleanup have been implemented; remaining items tracked below.

## Executive Summary

`generic_app` is a well-structured modular monolith (FastAPI + React) with meaningful recent investment in RAG (pgvector), memory (Mem0), caching (`backend/lib/*_cache.py`), pagination, and observability. Recent fixes include prompt-version resolution, background evaluation runs, non-blocking auth/settings I/O, durable RAG indexing via object storage, batched pgvector writes, bounded/paginated session/calendar/memory queries, and performance tuning (HTTP client reuse, provider retries, evaluation bulk insert, concurrent memory writes).

Remaining high-impact work:

1. **Unify RAG context assembly** across agent, `/ai/runs`, and `/rag/ask`.
2. **Extend degradation flags** to agent runs and `/ai/runs` (RAG `/retrieve` and `/ask` now expose structured outcomes).
3. **Consolidate orchestration** (`agent_service`, `prompt_context_builder`, `rag_answer_service`) behind one composed layer.

Overall state: **solid foundation, medium technical debt**. AI/RAG quality can improve incrementally (reranking, unified prompt assembly) without a rewrite.

## Priority Improvements

### 1. Unify retrieval-augmented context assembly
- **Category:** Architecture / AI
- **Severity:** High
- **Effort:** Large
- **Risk:** Medium
- **Files or modules affected:** `backend/modules/ai/application/prompt_context_builder.py`, `backend/modules/rag/application/rag_answer_service.py`, `backend/modules/rag/application/rag_context_builder.py`, `backend/modules/ai/service.py` (`run_prompt`, `retrieve_chunks`), `backend/modules/ai/models.py` (`AiDocument`, `AiDocumentChunk`)
- **Current problem:** Three divergent paths (agent, `/ai/runs`, `/rag/ask`) format context differently. Legacy `/ai/documents*` still delegates through `LegacyAiDocumentService` when RAG is enabled.
- **Recommended solution:** Single `PromptContextService`; route all RAG-enabled retrieval through `RagContextBuilder`; converge agent and run endpoints on the same assembly contract.
- **Expected benefit:** Fewer bugs, easier testing, clearer module ownership.

### 2. Extend structured degradation to agent and legacy run paths
- **Category:** AI / Reliability
- **Severity:** Medium
- **Effort:** Medium
- **Risk:** Low
- **Files or modules affected:** `backend/modules/ai/application/agent_service.py`, `backend/modules/ai/router.py`, agent API schemas
- **Current problem:** RAG `/retrieve` and `/ask` now return `RetrievalOutcome` / degradation flags; agent runs and `/ai/runs` still hide memory/retrieval failures behind empty context.
- **Recommended solution:** Reuse `RetrievalOutcome` and `memory_degraded` in agent responses; metric counters on agent path.
- **Expected benefit:** Consistent observability across all LLM entry points.

**Done (RAG ask/retrieve):** `RetrievalOutcome`, `/rag/retrieve` wrapper response, `/rag/ask` degradation fields, Prometheus counters (`rag_retrieval_degraded_total`, `rag_retrieval_no_match_total`, `rag_injection_chunks_filtered_total`, `rag_json_fallback_total`).

## Performance and Speed Improvements

### Heavy operations on request/response paths

| Location | Problem | Severity | Fix | Risk |
|----------|---------|----------|-----|------|
| `backend/modules/rag/api/routes.py` — `ask_rag` | Retrieve + memory + LLM + query logging in one request | High | Optional async job or streaming; at minimum strict timeouts | Medium |

| `backend/api/main.py` — `lifespan()` + `platform/service.py` — `ensure_defaults()` | Many sequential DB round-trips on every API worker boot | Medium | One-time seed migration or bulk upsert | Low |

| Location | Problem | Severity | Fix | Risk |
|----------|---------|----------|-----|------|

## Caching Improvements

### Existing cache infrastructure (keep and extend)

| Cache | Location | TTL | Invalidation |
|-------|----------|-----|--------------|
| Platform config, flags, email templates | `backend/core/cache.py` + `platform/service.py` | Configurable via settings | `invalidate_platform_cache()` on writes |
| Settings DB rows | `settings/service.py` | Redis + local TTL | Key delete on mutation |
| Observability status | `observability/service.py` | Short TTL | Time-based |
| Embeddings (indexing) | `backend/lib/embedding_cache.py` | 600s | TTL; skip texts > 4000 chars; keys include model + dimensions |
| Query embeddings (retrieval) | `backend/lib/embedding_cache.py` via `embed_retrieval_queries` | 3600s | TTL; model + dimensions in key |
| Retrieval results | `backend/lib/retrieval_cache.py` | 180s | `invalidate_retrieval_cache_for_document()` on index/delete |
| Memory search | `backend/lib/memory_search_cache.py` | 60s | Per-user invalidation on writes |
| User profile snapshot | `backend/lib/resource_cache.py` + `profile/service.py` | 90s | Invalidate on profile/avatar mutations |
| Project list | `backend/lib/resource_cache.py` + `projects/service.py` | 30s | Invalidate on project create (per user) |
| Calendar items | `backend/lib/resource_cache.py` + `calendar/service.py` | 60s | Invalidate on calendar/task due-date mutations |
| User directory | `backend/lib/resource_cache.py` + `users/service.py` | 60s | Invalidate on user profile name updates |

### Remaining recommendations

| What to cache | Notes |
|---------------|-------|
| Subscription plans list | Small table; OK uncached |
| Notifications list | Real-time expectations — avoid long TTL |
| Static platform metadata CDN | Example nginx snippet in `infra/nginx/frontend.conf.example`; use short TTL or ETag for flags |

### Cache invalidation

- **`cache_delete_pattern`:** Now deletes in batches during `scan_iter` to reduce large single deletes. Key versioning for retrieval keys remains optional if wildcard scans become hot.
- **Local + Redis dual layer:** Invalidation clears both layers via `cache_delete` (unchanged).

### Frontend alignment

- `frontend/src/config/queryTiming.ts` — `userProfile` 90s, `projects` 30s, `calendar`/`userDirectory` 60s, `platformMetadata` 300s; notifications keep 30s stale + 60s poll.
